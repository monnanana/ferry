# 叙事资源管理游戏开发方法论

> 适用类型：**Choice-based Survival Narrative**（互动叙事 + 资源生存游戏）
> 首次沉淀自：《最后的摆渡人》开发复盘，2026-04-21

---

## 一、游戏类型定义

**中文名**：互动叙事 + 资源生存游戏
**英文名**：Choice-based Survival Narrative

同时融合以下子类型：
| 子类型 | 核心特征 |
|--------|---------|
| 文字冒险（Text Adventure） | 核心交互是文字 + 选项 |
| 资源管理（Resource Management） | 多维资源要维持平衡 |
| 道德选择（Moral Choice） | 选择影响结局和角色关系 |
| Roguelite 叙事 | 多角色路线 + 不同周目体验 |

### 同类优秀参考作品
| 作品 | 学习要点 |
|------|---------|
| 80 Days | 时间资源管理 + 全球路线选择 |
| 奥太线过境（Papers Please）| 道德压力 + 规则与人性对立 |
| 行尸走肉（Telltale）| 延迟反馈型道德选择 |
| 俄勒冈小道 | 资源耗尽的紧张节奏 |
| Wildermyth | 角色个性化叙事 + 多结局 |

## 二、开局对齐文档（先于代码）

动代码前必须写清楚的 6 件事：

```
1. 资源维度：几个？边界值各是多少？
2. 场景节点：最少选项数（建议 ≥ 2 个有意义分支，不是「继续/什么都不做」）
3. 角色差异：是机制层差异还是纯数值微调（机制层 >> 数值微调）
4. 道德选择：几个？是否有「延迟反馈」（现在选，后面才看到代价）
5. 结局分叉：数值结局 + choices 旗帜位两种结合
6. 平台需求：是否需要手机适配（「发给朋友玩」= 移动端，要提前做）
```

## 四、「唯一负责人」原则

**每类操作只能由一处代码执行**，不能分散：

| 操作 | 唯一负责人 | 其他地方的规则 |
|------|-----------|--------------|
| 资源扣除 | `flyTransit()` | 场景函数禁止直接扣 |
| 死亡检测 | `ended()` 守卫 | 每次 modHP 后必须检查 |
| 场景渲染 | 各 `scene_xxx()` 函数 | 只做叙事，不做资源操作 |
| 结局判定 | `calcEnding()` | 不在场景函数里写结局逻辑 |

> ⚠️ 违反此原则 = 双扣 bug + 数字不符 + 多轮反复调试

## 五、上线前自检清单

```
□ runSimulation() 通过率 > 60%（自动跑 20 条路径验证平衡性）
□ 最穷路线能否走完（金钱下限验证）
□ 每个角色至少人工测一遍主路线
□ 死亡路线：体力/时间/警戒归零后，是否正确停止执行
□ 所有结局是否可达（特别是最好结局和最差结局）
□ 手机端截图确认（iPhone Safari + 微信内置浏览器）
□ 每个场景节点：有效选项是否 ≥ 2 个
□ 道德选择：是否有至少 1 个「延迟反馈」节点
```

## 六、常见踩坑 & 预防

| 坑 | 原因 | 预防方法 |
|----|------|---------|
| 死亡后继续执行 | 没有 `return` 中断 | 所有 `modHP` 之后加守卫 `if(...) return` |
| 资源双扣 | 两处代码都在扣同一资源 | 定义唯一负责人，其余处禁止扣 |
| 预告数字与实际不符 | 扣除逻辑分散在多处 | 统一由 `flyTransit` 负责 |
| 选项只能「继续」 | 场景设计时没设最低分支数 | 开局对齐文档中写死「≥2 有效选项」 |
| 角色差异感不强 | 只做数值微调 | 每个角色必须有独特**行为机制**，不只是数值 |
| UI 需求末期返工 | 响应式没提前考虑 | 开局确认「是否发给朋友玩」= 移动端适配 |
| 平衡性依赖人工试玩 | 路径太多人工覆盖不完 | 写 `runSimulation()` 自动跑所有路径 |

## 七、设计心法

1. **资源维度 3~5 个最佳**：太少无聊，太多烧脑，玩家记不住
2. **选项要有「代价差异」**：花钱省时间 vs 花时间省钱，而不是「继续/什么都不做」
3. **道德选择要有「延迟反馈」**：现在的选择，后面两个场景才显现代价，压迫感更强
4. **反转时机**：第一个反转放在中段（不能太早），最大反转放在结局前
5. **角色差异是机制不是数值**：「能用证件过检查」 >> 「体力+5」
